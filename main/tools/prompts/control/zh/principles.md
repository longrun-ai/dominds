## 核心概念

### 1. 提醒（Reminder）

提醒是临时工作集，但现在分成两个 scope：

- `dialog`：仅当前对话可见
- `personal`：在所有由你主理的后续对话里都可见

默认用 `dialog`。只有当这条提醒在所有由你主理的后续对话里、继续承担相同职责时仍然应该被看到，才升级为 `personal`。

**特点：**

- `dialog` 提醒只留在当前对话
- `personal` 提醒会在所有由你主理的后续对话里继续可见
- 可随时添加、修改、删除
- 默认应保持少量、可扫读、可直接指导下一步
- 准备 `clear_mind` 时，主线对话先把应由下一程知会的未落文档讨论细节补进差遣牒，再压缩成结构化接续包提醒项；支线对话直接维护足够详尽的接续包提醒项。若当前程已被系统置于吃紧/告急处置态，支线提醒项长度没有技术限制，也可先保留多条粗略提醒项过桥

**与 memory 的区别：**
| 特性 | dialog reminder | personal reminder | personal memory |
|------|-----------------|-------------------|-----------------|
| 持久性 | 仅当前对话 | 跨所有由你主理的后续对话 | 长期 / 文件持久化 |
| 可见性 | 当前对话 | 你当前可见的提醒范围 | 当前智能体 |
| 最适合 | 当前下一步、阻塞、易丢线索 | 职责相关的操作提示 | 稳定事实 / 可复用知识 |

### 1.1 Scope 选用原则

- 下列情况用 `personal`：
  - 这个 agent 在类似职责下反复要用到的首选命令
  - 这个 agent 持续要执行的安全检查
  - 应该跨对话保留的职责相关注意点
- 其余一律用 `dialog`：
  - 当前阻塞
  - 临时路径、id、命令、样例输入
  - 只对这次对话 / 当前程有用的过桥备注
- 如果内容本质上是长期稳定事实或知识资产，而不是当前工作集提示，它更可能属于 `personal_memory`，而不是 reminder。

### 2. 差遣牒

差遣牒是**任务契约**，也是任务执行期的**全队共享真相载体**；其中 `progress` 承担的是**准实时全队任务公告牌 / 当前有效状态快照**职责。

**结构：**

- **goals.md**：任务目标
- **constraints.md**：约束条件
- **progress.md**：面向全队同步的当前有效状态、关键决策、下一步与仍成立阻塞

**更新规则：**

- `mind_more` 用于少量追加条目；`change_mind` 用于整章替换
- 不重置对话轮次
- 变更对所有队友可见
- 写入 `progress` 时，应默认假设全队成员会用它快速同步“当前任务真相”，而不是阅读你的个人过程记录

## 跨对话回复路由

### 决策规则

- 当前支线未完成时，先判断团队规程 / SOP / 职责卡能否明确负责人；若能明确且属于执行性处理，直接 `tellask` / `tellaskSessionless` 对应负责人
- 只有当必须向诉请者补需求、澄清目标、裁决取舍、确认验收口径、提供缺失输入，或现有规程无法明确判责时：调用 `tellaskBack({ tellaskContent })`
- 需要人类亲自登录 / GUI / 验证码 / 高风险授权时：调用 `askHuman({ tellaskContent })`
- 当前支线已经完成，且当前 assignment 明确要求 `replyTellask`：调用 `replyTellask({ replyContent })`
- 当前支线已经完成，且当前 assignment 明确要求 `replyTellaskSessionless`：调用 `replyTellaskSessionless({ replyContent })`
- 当前是在回复一条诉请者发来的 `tellaskBack` 续诉请，且 runtime 暴露了 `replyTellaskBack`：调用 `replyTellaskBack({ replyContent })`
- 普通文本不是跨对话完成交付通道；如果你直接输出正文而没调 reply 工具，runtime 可能临时插入一条 `role=user` 的提醒要求你改用正确的 reply 函数

### 心智负担最小化规则

- 先专注把当前任务做对；`reply*` 只在最终对诉请者交付时用于收口
- 不要靠记忆硬选 reply 变体；以当前 assignment 头部和 runtime 当前暴露的函数名为准
- `reply*` 函数自身说明文案故意保持极简，只承载最小规格；情景判断看本手册的 principles / scenarios
- 若 runtime 只暴露一个 `reply*`，那就是当前应调用的唯一完成路径
- `tellaskBack` 只在“无法按现有规程明确判责，或必须回问诉请者”时成立，不是所有阻塞的默认第一动作

## 最佳实践

### 1. 提醒使用场景

- **当前工作集**：当前下一步、阻塞、关键定位信息
- **易丢细节**：临时路径、id、命令、样例输入
- **换程接续**：在 `clear_mind` 前整理接续包；若当前程已被系统置于吃紧/告急处置态，可先保留多条粗略提醒项
- **职责延续提示**：如果在所有由你主理的后续对话里也应该继续看到它，用 `personal`

### 2. 差遣牒更新时机

- **目标变更**：当任务目标发生变化时
- **约束调整**：当约束条件需要调整时
- **进度更新**：当需要向全队同步当前有效状态、关键决策、下一步或仍成立阻塞时

### 3. 更新策略

- 保持简洁：默认提醒项常见 1–3 条，优先 `update_reminder` 压缩/合并
- 区分载体：需要向全队同步的当前有效状态、关键决策、下一步与仍成立阻塞，写入 `progress` 这一准实时任务公告牌；提醒项只留个人/当前对话恢复所需细节
- 面向全队：`progress` 应保持可扫读、以“当前仍有效”为准，不要退化成个人日志、流水账、临时便签或历史残影堆积；少量新增用 `mind_more`，需要清旧/重排/压缩时用 `change_mind`
- 换程前收束：主线对话先把尚未落实到文档、且下一程需要知会的讨论细节写入差遣牒合适章节，再整理结构化接续包提醒项；支线对话不要维护差遣牒，也不要整理差遣牒更新提案，直接维护足够详尽的接续包提醒项。若系统已把当前程切到吃紧/告急处置态，则先保留多条粗略提醒项过桥也可以；当前程只做落文档/保信息 + `clear_mind`（支线只做保信息 + `clear_mind`），系统真正开启新一程后第一步再收敛
- 拒绝原料堆积：不要把长日志/大段 tool output 直接塞进提醒项

### 4. `progress` 内容取舍

- 适合写进 `progress`：
  - 当前已经生效的关键决策
  - 当前已确认、仍成立的 blocker
  - 当前全队应共识的下一步
  - 当前阶段性闭环与尚未闭环的 gap
- 不适合写进 `progress`：
  - “我刚刚看了什么文件”
  - “我准备先试一个小想法”
  - 只对当前说话者自己有用的临时便签
  - 无法判断是否仍然有效的历史痕迹

## 限制与注意事项

1. `dialog` 提醒会随对话结束而结束；`personal` 提醒会在所有由你主理的后续对话里继续可见
2. 差遣牒少量新增可用 `mind_more` 追加；整段替换用 `change_mind`，请确保合并已有内容
3. `mind_more` / `change_mind` 不重置对话轮次
4. 接续包提醒项只保留差遣牒仍未覆盖、但恢复工作容易丢的细节；主线对话应先把当前对话历史中应由下一程知会的未落文档讨论细节写入差遣牒合适章节；支线对话在吃紧/告急时只维护足够详尽的接续包提醒项
5. 不要把 `personal` 提醒堆成长期事实仓库；耐久知识应迁到 `personal_memory`
